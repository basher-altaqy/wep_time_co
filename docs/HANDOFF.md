## 1) ملخّص في فقرة واحدة

**Saff** (الاسم الداخلي للمشروع، اسم المستودع `wep_time_co`) منصّة عربية لتشكيل وتشغيل فرق تنفيذية صغيرة (2–10 أشخاص) حول فرصة أو مشروع، عبر نظام طلبات انضمام مُهيكَل بنمطَين (Capacity للساعات، Outcome للمخرجات). الرؤية الكاملة في `VISION.md` تَتجاوز ذلك إلى نظام تشغيل متكامل (Studio + Mission Canvas + Climb Map + Contribution Ledger)، لكنّ النسخة الحالية (v0.1) تَقتصر على الطبقة الأولى: Auth + Ventures + Applications. النيّة Lean MVP لاختبار فرضية واحدة قبل التَوسّع.

## 6) القرارات المعمارية الجوهرية

اتُّخذت عدّة قرارات يَجب أن تَفهم منطقها قبل التَفكير في تَغييرها.

**القرار 1: Supabase Auth لا Clerk**.
رغم أنّ تَحليلًا استراتيجيًّا أَوصى بـ Clerk بسبب دعمه للـ Organizations والـ Roles (مفيدة لـ Studio في v1)، اخترنا الإبقاء على Supabase Auth وبناء العضوية يدويًّا في جدول `studio_members` مع RLS تَفحص العضوية. السبب أنّ التحوّل إلى Clerk يَستلزم إعادة بناء جوهرية، بينما Supabase + RLS تُحقّق نفس الهدف. Clerk يَبقى خيارًا مفتوحًا لو وَصلنا مقياسًا يَستلزم ميزاته المتقدّمة (SSO، SAML).

**القرار 2: App Router لا Pages Router**.
لاستفادة من Server Actions التي تُبسّط الكثير، وتَتكامل مع cookies الجديدة في Next.js 15.

**القرار 3: Server Components افتراضيًّا**.
كل المكوّنات Server Components، إلا تلك التي تَستلزم تَفاعلًا (forms، أزرار interactive). يُمكنك التَعرّف على هذه عبر `"use client"` في رأس الملفّ.

**القرار 4: لا shadcn/ui ولا أيّ component library**.
بنينا 6 مكوّنات UI من الصفر لأنّ shadcn CLI يَفترض LTR، والمكتبات الأخرى ثقيلة أو غير مُحسَّنة للعربية. هذا قرار قد تُعيد النظر فيه في v0.2، لكنّ المكوّنات الحالية تَخدم المرحلة جيّدًا.

**القرار 5: Zod هو مصدر الحقيقة للتَحقّق**.
نفس الـ schema يُستخدم على الخادم في Server Actions وفي React Hook Form على العميل. اعتمد هذا النمط في أيّ كود جديد.

**القرار 6: السلاج (slug) عربي → لاتيني عبر transliteration بسيطة + nanoid(6)**.
موجود في `src/lib/utils.ts`. مثال: "منصّة التَعاون" → `mnst-altan-x9f2k1`. ليس مثاليًّا لكنّه يَكفي للـ MVP.

**القرار 7: RLS مع `force row level security` على كل الجداول**.
الـ default deny صارم. كل policy تُكتب صراحةً. كل query في Server Actions يَمرّ عبر JWT المستخدم. **لا تَستخدم Service Role Key في أيّ كود يَقترب من العميل**. هذا غير قابل للتفاوض.

## 7) ما يَنقص الـ MVP الحالي (قائمة بَسيطة)

هذه قائمة بأمور تَعرف عنها من الآن قبل أن تَكتشفها لاحقًا:

أوّلًا، **توليد types قاعدة البيانات**. الملفّ `src/types/database.ts` كَتبتُه يدويًّا كموقّت. يَجب توليده آليًّا بعد تَطبيق المهاجرات الفعلي بـ:
```
npm run db:types
```
أو يدويًّا:
```
supabase gen types typescript --linked > src/types/database.ts
```

ثانيًا، **next.config.mjs يَستخدم `experimental.typedRoutes`** الذي انتقل خارج experimental في Next.js 15. غيّره إلى `typedRoutes` على المستوى الأعلى من الإعدادات.

ثالثًا، **صفحة `/applications` في الـ Server Component تَستخدم join بصياغة Supabase**. قد يَحتاج التَأكّد أنّ الـ types مولَّدة صحيحًا حتى لا تَشتكي TypeScript. الكود يَعمل function-wise لكنّ الأنواع قد تَحتاج لإصلاح.

رابعًا، **لا توجد صفحة Edit Venture**. أُجِّلت ضمن US-3.4. أَضِفها في `src/app/(app)/ventures/[id]/edit/page.tsx`. المنطق نسخة من `new/` مع تَحميل القيم الموجودة والـ Server Action للتحديث.

خامسًا، **استرجاع كلمة السرّ** غير مُنفَّذ. صفحة `/login` تَحوي رابطًا معطّلًا "نسيت كلمة السر؟ (قريبًا)". أَضِف صفحة `/forgot-password` و route handler يَستخدم Supabase resetPasswordForEmail.

سادسًا، **تَأكيد البريد** يَعمل عبر `auth/callback/route.ts`، لكن للتَطوير المحلّي يُفضَّل تَعطيل تَأكيد البريد من لوحة Supabase تحت Authentication → Providers → Email → "Confirm email" = OFF.

سابعًا، **لا توجد اختبارات (tests)**. لا Playwright، لا Jest، لا Vitest. هذا قرار واعٍ للـ MVP، لكنّ v0.2 يَجب أن يَبدأ بإضافة Playwright لاختبار 3 سيناريوهات أساسية على الأقلّ (Signup→Create Venture، Login→Apply، Founder→Decide).

## 9) تَحذيرات معروفة من تَجارب الإعداد

اكتُشف ما يَلي خلال محاولة إعداد البيئة على ويندوز 11 (لم تَكتمل):

**تَحذير 1**: حزمة `supabase` (Supabase CLI كـ dev dependency) قد تَفشل أحيانًا في استخراج binary على ويندوز بسبب مضادّ الفيروسات أو Windows Defender. ستَرى تَحذيرًا:
```
npm warn tar TAR_ENTRY_ERROR ENOENT ... cli-windows-x64\package.json
```
الحلّ: ثبّت Supabase CLI بطريقة مستقلّة (Scoop أو تَنزيل المُنفّذ من GitHub Releases) بدل الاعتماد على npm package. أو استخدم لوحة Supabase Web Dashboard لتَطبيق المهاجرات يدويًّا.

**تَحذير 2**: `npm start` ليس الأمر الصحيح للتطوير. استخدم `npm run dev`. السبب: `npm start` يُنفّذ `next start` (وضع الإنتاج بعد build)، بينما `npm run dev` يُنفّذ `next dev` (وضع التطوير مع Hot Reload).

**تَحذير 3**: على ويندوز، المسارات العربية في Path تَعمل عمومًا، لكن قد تَظهر مشاكل ترميز في PowerShell القديم. استخدم Windows Terminal الحديث، أو cmd بترميز UTF-8 (chcp 65001).

**تَحذير 4**: إذا كانت لديك خوادم تَطوير قديمة لم تُغلق، ستَأكل منافذ 3000–3010. نَظِّف بـ:
```
taskkill /F /IM node.exe
```
(على ويندوز)
أو:
```
pkill -f node
```
(على Linux/Mac)

## 10) كيفية التَواصل مع المالك

المالك (Basher) ليس مهندسًا متخصّصًا، بل مؤسّس. تَواصل معه بهذه القواعد:

- **اشرح القرارات بالنتائج لا بالتَفاصيل التَقنية**. لا تَقل "غيّرت Next.js config"، قل "أَصلحتُ تَحذيرًا كان يَظهر عند بدء التشغيل".
- **عند الحاجة لقرار، اعرض خيارَين بأثرهما الواضح، لا أكثر**.
- **التَوثيق بالعربية مُفضَّل عند الحديث معه**، لكنّ الكود ورسائل commit بالإنجليزية كما اعتدنا.
- **احترم رؤيته**. الرؤية الكاملة في `VISION.md` تَستحقّ الدفاع عنها. لو رأيتَ مشكلة في الرؤية، ناقشه فيها بِحجج مَنطقية، لا تَغيّرها من جانبك.

## 11) ما لا يَجب فعله

ثلاثة أمور تُفسد المشروع لو وَقعتَ فيها:

أوّلًا، **لا تَستخدم `SUPABASE_SERVICE_ROLE_KEY` خارج Server Actions موثوقة**. لو وَجدتَ إغراءً، توقّف وفكّر مرّتَين. الـ Service Role يَتجاوز RLS، فلو سَرَب إلى العميل، كل البيانات مكشوفة.

ثانيًا، **لا تَبني v1 (Studio + Canvas) قبل التَحقّق من v0.1**. الإغراء سيَكون قويًّا لأنّ v1 أَجمل من الناحية البصرية، لكنّك ستَبني فوق أساس لم يُختبَر، وستَدفع الثمن مضاعفًا.

ثالثًا، **لا تَدمج اعتمادات غير ضرورية**. الـ MVP خفيف عمدًا. كل حزمة جديدة تُضاف يَجب أن يَكون لها مبرّر صريح، وأن تُسجَّل في issue قبل إضافتها.
